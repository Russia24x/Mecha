   371→
   372→**Design intent** (DESIGN_PILLARS.md lines 10-25, CREATIVE_DIRECTOR_REVIEW.md lines 49-65):
   373→> *"Heavy: cannot cancel mid-animation. Precise: clear telegraph. Punishing: mistake = damage = quick death, but fair."*
   374→
   375→**What IS implemented:**
   376→- ✅ Telegraphs per enemy type with distinct visual styles (`EnemyEntity.ts` lines 210-256): drone=red circle, sniper=pulsing dot, flying_ai=amber ring above, heavy/elite=purple ground shake.

   379→- ✅ Flying AI dive-bombs (`EnemyEntity.ts` lines 328-336). Contrary to Game Director Review, flying_ai is functional.
   380→- ✅ Contact damage + knockback on player hit (`GameScene.ts` lines 804-815).
   381→- ✅ Hit-stop, particles, screen shake on hits (via `CombatSystem`, `ParticleSystem`).
   382→
   383→**What's broken or missing:**
   384→- ❌ **No animation commitment.** `PlayerEntity.ts` `tryFire` (line 466), `tryMelee` (line 528), `tryDash` (line 440) only check cooldowns. Movement is fully controllable during fire/melee. You can sprint, jump, dash, and shoot simultaneously with no lock. This violates "Heavy" pillar #1.

   390→
   391→**Verdict:** ❌ Combat is responsive but weightless. You can fire+run+dash+jump simultaneously, death is a slap on the wrist, and there's no posture system to reward aggression or patience. This is the second-biggest gap.

  1635→- Re-investigated the actual race condition with debug logs in flushToIndexedDB.
  1636→  Found the REAL root cause: NOT a simulator difference, but a genuine race:
  1637→  1. beforeunload → flushIfDirty → doFlush → saveInFlight=true → await flushToIndexedDB()
  1638→  2. flushToIndexedDB: await writeProfileData(slotId, this.cache)  ← multiple awaits inside
  1639→  3. DURING these awaits, setQuestFlag('stop_test', true) runs → mutates this.cache, sets dirty=true
  1640→  4. writeProfileData eventually completes, writes this.cache (which NOW has stop_test)
  1641→  5. flushToIndexedDB: this.dirty = false  ← OVERWRITES the dirty=true from step 3!
  1642→  6. stop() sees dirty=false, flushIfDirty returns without writing
  1643→  7. Result: stop_test is in this.cache but never persisted
  1644→
  1645→  This race is REAL in both fake-indexeddb AND real browsers. It's caused by
  1646→  the async gap between writeProfileData start and dirty=false assignment.

  1649→  1. Snapshot the cache BEFORE writing (JSON deep clone)
  1650→  2. Set dirty=false BEFORE the write (not after)

  1652→  4. After write completes, if dirty was re-set, leave it — next flush picks it up
  1653→
